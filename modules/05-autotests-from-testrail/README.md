# Инструкция для Cursor: Написание автотестов Playwright на основе тест-кейсов TestRail

## Быстрый старт (копипаст)

**Вариант A — автотесты по секции TestRail:**
```
Открой и следуй: modules/05-autotests-from-testrail/README.md

Создай автотесты на тесты секции <sectionId>.
Спек, куда писать: <путь к .spec.ts>
```

**Вариант B — автотесты по выбранным кейсам:**
```
Открой и следуй: modules/05-autotests-from-testrail/README.md

Допиши автотесты на кейсы TestRail: <caseId1>, <caseId2>, ...
Спек, куда писать: <путь к .spec.ts>
```

## Входные данные (что даёт пользователь)

- **Обязательно:** откуда брать кейсы (секция — `sectionId`, или список кейсов — `caseId1, caseId2, ...`) и **путь к файлу спеки**, куда писать тесты (например `playwright/tests/LeftPanel/trace-list.spec.ts` или файл в репозитории my-assistant).

Данные по кейсам (title, steps, expected result) ты получаешь **из TestRail** через MCP: `get_cases` с **project_id: 5** и `section_id` (или по списку id кейсов). По проекту кода ничего не ищешь — используешь только то, что дал пользователь, и кейсы из TestRail. 

## Принцип работы модуля

Ты действуешь по шагам из этого README. Пользователь даёт команду и входные данные — ты получаешь кейсы из TestRail и генерируешь тесты в указанный файл.

## Архитектурные решения (при написании автотестов)

**Основа:** при создании и доработке автотестов соблюдать соглашения ниже (слои, локаторы, переиспользование, структура теста). Внешние файлы не открывать.

**Слои и ответственность**

- **locators** — только функции, возвращающие `Locator`; без кликов, ожиданий, ассертов и бизнес-логики. Новые селекторы сначала в locators, использование — через pages.
- **pages** — Page Objects и Widget Objects: пользовательские действия, сценарные шаги, ожидания. `page` считается внутренней деталью реализации; в spec при использовании PO запрещён прямой доступ к `page` и смешивание (часть шагов через PO, часть через `page.getBy...`). Все взаимодействия с UI — только через методы Page Object / Widget Object.
- **Spec** — только сценарий и проверки: вызовы Page Object, хелперов, `expect` по данным из PO/хелперов. В spec нет inline-селекторов, фабрик моков и инфраструктурных helper-функций.
- **helpers** — переиспользуемая orchestration-логика, setup-функции, проверочные утилиты (подготовка сцены, работа с файлами, эмуляция DnD, настройка моков, закрытие уведомлений). Assert-helpers при необходимости в helpers.
- **mock** — mock-данные, конфиги и фабрики данных; без сценарной orchestration. Настройка моков через хелперы, не inline в spec. В spec только вызов setup-функции, без больших inline-блоков.

**Правила работы с селекторами**

- В spec локаторы только через Page Object / locators (без inline `page.locator`, `page.getBy*` в спеке). Предпочтительно — `data-testid` и `getByTestId`; `getByRole` — fallback внутри контейнера с `data-testid`. Избегаем CSS-классов, меняющегося/локализованного текста, хрупких селекторов по индексам/порядку DOM. Если `data-testid` нет — пишем тест с временным fallback (в locators/PO с `TODO`), помечаем `MISSING data-testid`; задачу на разметку при необходимости оформляем по модулю 11. Локаторы не блокируют завершение теста.

**UI-тексты и константы**

- UI-строки (заголовки, тексты модалок, названия кнопок и т.д.) не хардкодим в pages и tests. Храним централизованно (например `helpers/constants/uiText.ts`) и импортируем по месту использования.

**Структура describe / test** (дополнение к solo-rtm для маппинга на TestRail)

- **describe** — по секциям TestRail или по смыслу (область UI + контекст).
- **test** — один сценарий на один кейс TestRail (или явно сгруппированный); в названии маппинг на кейс, например `[caseId]`.
- Имя теста: `ABC-### <TestRail case title> [<caseId>]` (префикс фичи 2–5 символов, точный title из TestRail, id кейса в квадратных скобках).
- Внутри теста — один или несколько `test.step` с явным AAA (Arrange → Act → Assert). Каждый step покрывает одну логическую фазу; ассерты группируем по смыслу. Формулировки Assert — про ожидание, не про реализацию. Если один атомарный сценарий с одной проверкой, `test.step` можно не добавлять.

**Переиспользование и организация**

- **Сначала — переиспользовать.** Новые методы/хелперы писать только когда нужной логики действительно ещё нет в pages или helpers. Перед добавлением кода проверить: есть ли уже метод открытия экрана, раскрытия узла, настройки мока, проверки видимости и т.п.; если есть — вызывать его, не дублировать.
- **Дублирования избегать в любом виде:** и явного (одинаковый код в нескольких спеках), и скрытого (та же логика, но переписанная своими словами, с другим порядком вызовов или с небольшими отличиями). Если логика повторяется — вынести в page/helper и вызывать оттуда.
- Повторяющиеся шаги — в pages или helpers. Фикстуры: `import { test, expect } from '../../fixtures'` (или корректный relative path). TypeScript: не использовать `any`; типизировать фикстуры/моки, при необходимости `satisfies`.

**Стабильность**

- Тест не зависит от порядка выполнения других тестов; при необходимости каждый тест сам подготавливает сцену и моки. Явные ожидания: `expect(...).toBeVisible()`, `waitForLoadState('domcontentloaded')`, ожидание через PO, без жёстких `waitForTimeout`.

## 1. Алгоритм работы

### Что даёт пользователь

- **Вариант A:** `sectionId` секции TestRail + путь к файлу спеки. Нужно получить все кейсы этой секции и написать по ним автотесты.
- **Вариант B:** список id кейсов TestRail (`caseId1, caseId2, ...`) + путь к файлу спеки. Нужно получить эти кейсы и дописать автотесты в указанный файл.

Путь к спеке — обязателен (например `playwright/tests/LeftPanel/trace-list.spec.ts`). По этому файлу видно существующие импорты, pages, helpers — переиспользуй их, новое пиши только при отсутствии нужного.

### Шаг 1 — Получение кейсов из TestRail

Получи данные кейсов через **MCP TestRail**:
- **Вариант A** (по секции): `get_cases` с `project_id: 5` и `section_id`. **project_id всегда 5** (DrillSpot/RTM).
- **Вариант B** (по выбранным кейсам): для каждого id из списка вызови `get_case` с параметром `case_id` (например, `get_case` с `case_id: 429420`). Один вызов — один кейс.

Остальные параметры (suite_id и т.д.) — в `testrail-field-ids.md` в корне проекта или уточни у пользователя.

Из каждого кейса нужны: id, title, шаги (steps) — поле `custom_steps_separated` или аналог, expected result, при необходимости — теги/лейблы, прекондиции (`custom_preconds`). Никакого поиска по коду проекта: только TestRail + путь к спеке от пользователя.

### Шаг 2 — Анализ кейсов и контекст

Изучи полученные кейсы: шаги, ожидаемый результат, прекондиции. Учитывай **лейблы/теги** кейса (например `linear_widget`, `trace_list`, `track`) — они подсказывают, что должно быть на экране (дашборд, виджет, дерево). Если по кейсу неочевидно, что подготавливать — задай пользователю короткий уточняющий вопрос (например: «Нужен Linear widget на окне или окно пустое?»), не додумывай молча.

Если пользователь дал ссылку на аналитику — открой её для контекста. Если нет — достаточно текста кейсов.

### Шаг 3 — Написание автотестов

Промт для генерации автотестов:

Ты — QA Automation Engineer в проекте RTM. Сгенерируй Playwright E2E тест(ы) 

Цель: создать единообразные тесты, читаемые как спецификация, хорошо логирующиеся в HTML report/trace, и легко маппящиеся на TestRail.

Формат (ОБЯЗАТЕЛЬНО):
- `test.describe('<UI area> > <context>', () => { ... })`
  - строка describe = "область UI + контекст сценария", берём из TestRail (если есть).
- Имя теста: `ABC-### <TestRail case title> [<caseId>]`
  ABC — внутренний префикс фичи/модуля (2–5 символов), который используется для идентификации набора тестов и их нумерации внутри фичи.
Примеры: TL = Trace List, AUTH = Authentication, DASH = Dashboards, LP = Left Panel.
ABC-### — внутренняя нумерация тестов в рамках выбранной фичи, не связана с TestRail ID.
Пример: TL-001 ... [61960], TL-002 ... [61966].
  - `<TestRail case title>` — точный title из TestRail.
  - `[<caseId>]` — числовой id кейса из TestRail в квадратных скобках.
- Внутри теста — один или несколько `await test.step('<название>', async () => { ... })`. Разделение по AAA: отдельные шаги для Arrange, Act и Assert (или несколько Assert-шагов, если проверок много).
  - Название шага — короткая фраза про результат/действие (из Expected Result TestRail или смысл проверки). Удобно начинать с префикса: `Arrange: ...`, `Act: ...`, `Assert: ...`.
- **Содержимое шагов:**
  - **Arrange / Act** — только код: вызовы helpers, PO, setup моков, навигация, клики. Без комментариев-заголовков внутри шага.
  - **Assert** — проверки через `expect(...)`. Если в одном шаге несколько проверок — перед каждой можно комментарий `// Assert 1: <что проверяем>`, `// Assert 2: ...`, чтобы в отчёте было понятно, какая упала. Альтернатива: разнести по разным `test.step('Assert: ...', ...)` (один шаг — одна логическая группа проверок).
  - Формулировки Assert — про ОЖИДАНИЕ пользователя/системы, не про реализацию (например «модалка импорта не открывается», «таблица с данными видна», а не «noModalOpened === true»).
- **Пример структуры шагов** (по образцу):
  ```ts
  await test.step('Arrange: mock ..., open tab, expand tree', async () => {
    helper.setupBhaSuccessMock();
    await helper.openSectionsAndRunsTabDirect(...);
    await objectTree.expandWell(...);
    await objectTree.expandSectionChevronInTree(...);
  });
  await test.step('Act: drag and drop file onto Run node', async () => {
    noModalOpened = await objectTree.dropFileOnRunNode({ ... });
  });
  await test.step('Assert: import modal does not open (file sent directly)', async () => {
    expect(noModalOpened).toBe(true);
  });
  await test.step('Assert: components table visible with imported data', async () => {
    await expect(bhaTable.getTableContainer()).toBeVisible({ timeout: 10000 });
    await expect(bhaTable.getSuccessNotification()).toBeVisible({ timeout: 15000 });
    await expect(bhaTable.getComponentRows()).toHaveCount(MOCK_BHA_SUCCESS_COMPONENTS_COUNT);
  });
  ```
- Ассерты в автотесте должны соответствовать проверкам в тесте TestRail (expected/проверки из кейса должны быть отражены в asserts; если что-то не автоматизируется — это явно фиксируется)
- TypeScript:
  - НЕ использовать `any`
  - типизировать фикстуры/моки, где возможно использовать `satisfies` для массивов/объектов.
- Стабильность:
  - тест не должен зависеть от порядка данных/случайных таймингов
  - использовать `expect(...).toBeVisible()`/`waitForLoadState('domcontentloaded')`/`waitForURL` где оправдано.

Импорт/организация:
- Используй `import { test, expect } from '../../fixtures'` (или корректный relative path по месту файла).
- В spec не дублировать setup и локаторы: использовать Page Object и helpers (см. раздел «Архитектурные решения» выше). Вспомогательные функции (setup, assert-helpers) выноси в helpers (отдельный модуль) или в верх файла spec и документируй JSDoc (HOW/WHY). Локаторы — только в Page Object или в модуле локаторов.
- Не меняй существующие тесты; добавляй новый тест + новые helper'ы только при необходимости.


Сгенерируй:
- Готовый код теста(ов) в файл спеки, который указал пользователь (если файла нет — создай по этому пути).
- Один тест — один сценарий (один кейс TestRail); внутри теста один или несколько `test.step` с явным AAA.
- Внутри шагов: Arrange/Act — только код (helpers, PO, моки); Assert — `expect(...)`, при нескольких проверках в одном шаге — комментарии `// Assert 1: ...`, `// Assert 2: ...` или отдельный `test.step('Assert: ...')` на группу проверок (см. пример структуры шагов выше).
- Только код/дифф без лишних объяснений.

**Куда писать:** в файл спеки, который указал пользователь в команде (поле «Спек, куда писать»). Если пользователь не указал путь — сохраняй в `results/05-autotests-from-testrail` с осмысленным именем файла.
