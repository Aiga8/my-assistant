# Инструкция для Cursor: Создание баг-репортов (Jira)

## Быстрый старт (копипаст)

```
Открой и строго следуй инструкциям: modules/09-jira-bug-reports/README.md

Подготовь баг-репорт в Jira (я создам тикет сама) / Создай баг в Jira.

Контекст:
- Область: <Spreadsheet | Linear | EDM | ...>
- Шаги воспроизведения: <...>
- Ожидаемое поведение: <...>
- Фактическое поведение: <...>
- (Опционально: ссылки на скриншоты, Jira-задача, окружение)
```

---

## 0. Цель модуля

По описанию пользователя (шаги воспроизведения, ожидаемое/фактическое поведение, контекст) подготовить **баг-репорт** в формате Jira: Summary (заголовок) и Description (описание с разделами). По запросу пользователя — создать тикет в Jira через MCP (или только выдать текст для ручного создания).

---

## 1. Что изучить перед написанием

Если пользователь дал ссылки на Jira/Confluence/скриншоты — сначала изучи их.

Если дана Jira-задача (DS-..., QAA-...):
- прочитай description и комментарии
- выпиши факты: шаги, ожидание, факт, окружение

---

## 2. Как формировать Summary (заголовок)

- Кратко и однозначно: что сломано / при каких действиях.
- Префикс области по проекту: `Spreadsheet. ...`, `Linear. ...`, `EDM. ...`, `Drillspot. ...` и т.п.
- Избегай общих формулировок («не работает», «ошибка» без контекста).

---

## 3. Как формировать Description (описание)

Используй Jira-форматирование: `*Bold*`, `{{code}}`, `#` / `*` для списков.

Структура:

1) **Steps to Reproduce** (шаги воспроизведения)  
   Нумерованный список шагов. Без лишних деталей, но достаточно для воспроизведения.

2) **Expected Result**  
   Что должно происходить.

3) **Actual Result**  
   Что происходит на самом деле.

4) **Environment** (по возможности)  
   Браузер, ОС, версия приложения, тестовое окружение.

5) **Attachments**  
   Оставь пустым или укажи «скриншоты приложу» — пользователь добавит файлы.

Опционально:
- **Preconditions** — если важно (логин, проект, настройки).
- **Notes** — ссылка на связанную задачу, регрессия, частота (всегда / иногда).

---

## 4. Если не хватает данных

Не выдумывай. Если без уточнения можно ошибиться — задай 1–3 коротких вопроса. Если пользователь не отвечает — продолжай и явно перечисли *Assumptions* в конце описания.

---

## 5. Формат ответа

Верни:
1) **Summary**
2) **Description**

Если пользователь просит **создать тикет в Jira** — вызови MCP Atlassian (Jira), инструмент `jira_create_issue`. Перед вызовом проверь схему: `mcps/user-mcp-atlassian/tools/jira_create_issue.json`. Укажи `issue_type`: **Bug**, project_key и поля по правилам проекта (Labels, Component, при необходимости Parent/Epic). После создания верни ключ задачи и ссылку.

---

## 6. Сохранение черновика

Всегда сохраняй результат в:

`results/09-jira-bug-reports/`

Формат имени:

`dd_mm_yy_bug_<area>_<short>.md`
