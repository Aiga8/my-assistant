# Инструкция для Cursor: Подготовка Improvement (Jira DS) — заголовок + текст

## Быстрый старт (копипаст)

```
Открой и строго следуй инструкциям: modules/07-improvements/README.md

Подготовь improvement в Jira (я создам тикет сама).
Мне нужен:
1) Заголовок (Summary)
2) Текст (Description)

Контекст/ссылки:
- Область: <Spreadsheet | Linear | EDM | ...>
- Что сейчас: <кратко current behavior>
- Ожидаемое поведение: <кратко expected behavior>
- Jira (если есть): <DS-...>
```

---

## 0. Цель и стиль

Стиль:

- Summary — **кратко, по делу, на английском**, с префиксом области: `Spreadsheet. ...`, `Linear. ...`, `EDM. ...`
- Description — на русском, но структурно (разделы + буллеты), без воды
- Используй Jira-форматирование:
  - `*Bold*`
  - `{{inline code}}` для названий флагов/переменных/функций
  - нумерованный список: `#`
  - маркированный список: `*`

---

## 1. Что изучить перед написанием

Если пользователь дал ссылки на Jira/Confluence — сначала изучи их.

Если дана Jira-задача (DS-...):

- прочитай description
- прочитай комментарии (если есть полезные детали/кейсы)
- выпиши 3–7 фактов: current behavior, expected behavior, constraints

---

## 2. Как формировать Summary (заголовок)

Правила:

- начинай с области: `Spreadsheet.` / `Linear.` / `EDM.` / ...
- далее — действие + ожидаемое улучшение
- избегай двусмысленности и внутреннего жаргона, если он не общеупотребим в проекте

---

## 3. Как формировать Description (текст)

Требования:

- Пиши максимально **кратко и однозначно** (без “может/возможно/вероятно/как будто/непредсказуемо”).
- Не делай “отчёт” — только суть.
- Не используй отдельный блок “pain/impact/почему это проблема”.
- Не пиши `*Acceptance criteria:*` — обходимся без него.


Далее:

1) Блок описания (без заголовка `Description:` внутри текста)
   - что происходит сейчас (current behavior), 1–3 буллета
   - можно добавить 1–2 буллета “контекста”, как проявляется проблема в разных состояниях UI

2) `**Proposed Solution:**` (если есть)
   - что поменять
   - где примерно (UI/логика/ограничение/индикатор)

3) `**Expected Behavior:**` или `**Expected result:**`
   - что пользователь должен увидеть/получить после улучшения

4) `**Attachments:**` (оставь пустым, пользователь заполнит)

Опционально добавляй:

- `*Preconditions:*` (если важно)
- `*Notes / Constraints:*` (если есть фича-флаги, лимиты, версии и т.п.)

---

## 4. Если не хватает данных

Не выдумывай.

Если без уточнения можно ошибиться — задай 1–3 коротких вопроса.

Если пользователь не отвечает — продолжай, но явно перечисли *Assumptions* в конце описания.

---

## 5. Формат ответа (то, что нужно пользователю)

Верни строго 2 блока:

1) **Summary**
2) **Description**

И ничего лишнего (никаких дополнительных разделов вне структуры из пункта 3).

---

## 6. Сохранение черновика (обязательно)

Всегда сохраняй результат в:

`results/07-improvements/`

Формат имени:

`dd_mm_yy_improvement_<area>_<short>.md`
