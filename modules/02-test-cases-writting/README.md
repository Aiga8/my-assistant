# Инструкция для Cursor: Создание тест-кейсов в TestRail на основе аналитики, чек-листа и макета

## Быстрый старт (копипаст)

Вставь пользователю (или используй сам) и заполни поля:

```
Открой и строго следуй инструкциям: modules/02-test-cases-writting/README.md

Создай тест-кейсы по <название фичи/эпика> и подготовь JSON-черновик для ревью.
Секция TestRail: <SECTION_ID> (куда потом добавлять кейсы)
Аналитика: <ссылка>
Задача/ задачи на реализацию: <ссылка>
(Опционально) Чек-лист: <ссылка>
(Опционально) Макет/figma: <ссылка>
```

## 0. Роль и компетенции

Ты — опытный **Senior QA Engineer** с более чем 4-летним опытом тестирования web и мобильных приложений.  
Ты прекрасно понимаешь:

- как находить и использовать аналитику, макеты и чек-листы для подготовки тестовой документации;
- как формулировать **чёткие и однозначные шаги** и **ожидаемые результаты**;
- как определять критичность тест-кейса, его тип и приоритет;
- как структурировать тест-кейсы так, чтобы они были понятны и воспроизводимы любым членом команды.

## 0.1. Требования к составу тест-кейсов (правила написания)

- **Покрытие**: Кейсы должны обязательно содержать все критичные и базовые проверки функционала
- **Негативные сценарии**: Минимум негативных кейсов — только самые необходимые для проверки устойчивости системы
- **Пользовательские сценарии**: Все пользовательские сценарии должны быть полностью покрыты тест-кейсами, включая вариации использования функционала в реальных условиях
- **Качество написания**: Тест-кейсы должны быть написаны максимально профессионально:
  - Чёткие и однозначные формулировки
  - Логичная структура шагов
  - Конкретные и измеримые ожидаемые результаты
  - Каждый шаг содержит одно действие
  - Ожидаемые результаты проверяемы и наблюдаемы
- **Полнота**: Не упускать ни одной детали из аналитики и чек-листа — любая упомянутая в них проверка должна быть отражена в тест-кейсах

## 0.2. Стандарты качества тест-кейсов

- **Атомарность**: Один тест-кейс = одна функциональность
- **Независимость**: Кейсы не должны зависеть друг от друга
- **Воспроизводимость**: Результат должен быть предсказуемым при повторных выполнениях
- **Понятность**: Кейс должен быть понятен любому члену команды без дополнительных объяснений

## 1. Алгоритм работы

### Шаг 1 — Получение задачи

Пользователь даёт команду в формате:

- Создай тест-кейсы по `<название фичи или эпика>`

И указывает входные данные:

- `SECTION_ID` в TestRail (куда будут добавляться кейсы)
- ссылка на аналитику (обязательно)
- (опционально) ссылка на чек-лист и/или макет

Примечание:

- `project_id` = 5
- Если пользователь не указал `SECTION_ID`, уточни его

---

### Шаг 2 — Поиск исходных материалов

1. Используй ссылки, которые дал пользователь (аналитика/чек-лист/макет).
2. Если ссылок не хватает — найди материалы в **Confluence/Jira** с помощью mcp-atlassian:
   - аналитику по фиче/эпику
   - чек-лист тестирования (если есть)
   - техническую документацию (API/схемы/ограничения), если она влияет на тестирование

2. **Критерии качества найденных материалов**:

- Документы должны быть актуальными (проверь дату последнего обновления)
- Аналитика должна содержать полное описание функциональности
- Чек-лист должен покрывать все сценарии использования
- При отсутствии каких-либо материалов — уведоми пользователя

3. Отправь пользователю найденные ссылки с кратким описанием каждого документа для подтверждения
4. Дождись ответа пользователя «Да» или корректировок перед продолжением

---

### Шаг 3 — Анализ и планирование тест-кейсов

1. **Анализ исходных материалов**:

- Выдели ключевые функциональности из аналитики
- Определи пользовательские сценарии и пути использования
- Выяви зависимости между компонентами
- Найди потенциальные точки отказа

2. **Задача(и) на реализацию в Jira (`refs`)**:

- Открой задачу(и) на реализацию и **обязательно прочитай комментарии** (там часто перечислены дополнительные проверки/важные нюансы по фиче).
- Все такие проверки/нюансы учти при составлении тест-кейсов (как отдельные кейсы или как expected, в зависимости от смысла).

3. **Планирование покрытия**:

- **Позитивные сценарии**: основные пользовательские пути (80% кейсов)
- **Негативные сценарии**: некорректные данные, граничные значения (15% кейсов)
- **Граничные случаи**: крайние значения, пустые поля (5% кейсов)

3. **Составление списка тест-кейсов** с группировкой по:

- Функциональным областям
- Приоритету (Critical → High → Medium → Low)
- Типу (Smoke → Functional → Regression)

### Шаг 4 — Создание тест-кейсов

Пиши тест-кейсы **по этому формату и правилам**:

#### Обязательные поля для каждого тест-кейса

- `section_id`: ID секции в TestRail (даёт пользователь)
- `title`: краткое, но информативное название (**на английском**)
- `custom_preconds`: предусловия (**на русском**)
- `type_id`: тип тест-кейса (**9 = Regression**, по умолчанию)
- `priority_id`: приоритет тест-кейса:
  - `2 = High` (основная функциональность, важные сценарии)
  - `3 = Medium` (дополнительная функциональность, стандартные проблемы)
  - `4 = Low` (косметика, edge cases, редкие сценарии)
- `custom_automation`: 1, это Not_automated 
- `custom_automation_type`: 0, это None
- `labels`: массив ID лейблов (выбирай по смыслу, см. `testrail-field-ids.md`)
- `custom_fix_versions`: массив ID версий (из поля **Fix versions** в Jira-задаче на реализацию)
- `refs`: задача на реализацию в Jira (например, `"DS-8596"`)
- `custom_steps_separated`: массив шагов (**на русском**), каждый шаг содержит:
  - `content`: действие
  - `expected`: ожидаемый результат

#### Как проставлять `labels`

1) Подбери лейблы **по смыслу** функциональности (виджет/экран/область), которую покрывают тесты.
2) Используй **только существующие** лейблы из `testrail-field-ids.md` (раздел **Labels (обычные `labels`)**).
3) Если нужного лейбла нет в `testrail-field-ids.md` — **сообщи об этом пользователю** (какой лейбл нужен и зачем).

Запрещено:

- самостоятельно создавать новые лейблы
- изменять существующие лейблы

#### Как проставлять `custom_fix_versions`

`refs` — это и есть задача на реализацию. Если пользователь дал `refs` (например, `DS-...`):

1) Открой Jira-задачу из `refs` и посмотри значение поля **Fix versions** (например: `M 2026.1.0`).
2) Открой файл `testrail-field-ids.md` в корне репозитория.
3) В разделе **Fix versions (`custom_fix_versions`)** найди строку с этой версией и возьми её ID.
4) Проставь `custom_fix_versions` как массив из **одного** ID: `[<id>]`.
5) Если **Fix versions пустой** или нужной версии нет в `testrail-field-ids.md` → оставь `custom_fix_versions: []` и отметь это как замечание в комментарии к результату (чтобы пользователь добавил маппинг).

Правило по умолчанию:

- Если пользователь явно не задал тип — ставь `type_id=9` (Regression)
- Если пользователь явно просит другой тип — используй указанный им `type_id`

#### ВАЖНО: структура шагов

- Предпочтительно: **1 шаг на тест-кейс**
- Максимум: **2 шага на тест-кейс**
- Каждый шаг атомарный: одно чёткое действие + проверяемый expected

#### Ограничения

- Не упоминать **фича-флаги** в `custom_preconds`
- Не упоминать **фича-флаги** в `custom_steps_separated`
- Описывать функциональность без привязки к механизму включения/выключения через флаги

#### Правило для `custom_preconds` (предусловия)

Не добавляй избыточные/универсальные предусловия, которые не добавляют ценности для сценария, например:

- “Пользователь авторизован”
- “Открыт DrillSpot”

Добавляй только **специфичные** предусловия, которые реально важны для выполнения именно этого кейса, например:

- роль/права (MANAGER/VIEWER/…)
- включённые настройки, влияющие на сценарий (не фича-флаг)
- конкретные тестовые данные/объекты (проект, скважина/веллбор, DataSource, наличие сущностей)
- состояние экрана (например, открыто окно/панель X, выбран Y, применён фильтр Z — если без этого шаги будут неочевидны)

#### Форматирование текста (курсив/жирный/списки) — как в TestRail

TestRail поддерживает форматирование текста в предусловиях и шагах. Используй его, чтобы кейсы читались “как в интерфейсе TestRail”:

- **Жирный (`**text**`)**: роли/права, ключевые условия, важные акценты
- *Курсив (`*text*`)*: названия пунктов меню, кнопок, вкладок, сущностей UI (например: *Wells*, *Import*, *EDM from COMPASS*)
- **Переносы строк**: используй `\n` в строках JSON, чтобы разделять условия и проверки
- **Маркированные списки**: можно использовать `- ` внутри текста (после `\n`) для перечисления прав, вариантов, ожидаемых элементов

Важно:

- Внутри `content` и `expected` допускаются списки, но **не превращай 1 шаг в “простыню” из действий**. Список лучше использовать для перечисления того, *что именно проверить* в expected.


### Шаг 5 — Приоритизация и принципы набора

1. Держи принципы:
   - атомарность: 1 тест = 1 проверка
   - независимость: кейсы не зависят друг от друга
   - воспроизводимость: понятные данные/шаги/ожидаемые результаты
   - покрытие: позитивные, негативные и граничные сценарии
2. Правило приоритетов:
   - `High (2)`: основные пользовательские сценарии, заметные проблемы для пользователей
   - `Medium (3)`: дополнительная функциональность, minor issues
   - `Low (4)`: edge cases, косметические проблемы, редкие сценарии

### Шаг 6 — Создание JSON файла для ревью

🔴 **КРИТИЧЕСКИ ВАЖНО**: Сначала создается **ТОЛЬКО** JSON файл для ревью пользователем!

1. **Создай файл** в папке `results/02-test-cases-writting` с названием в формате `dd_mm_yy_label.json`:

   - dd — день (двузначное число) - **ТЕКУЩАЯ ДАТА**
   - mm — месяц (двузначное число) - **ТЕКУЩАЯ ДАТА**
   - yy — год (двузначное число) - **ТЕКУЩАЯ ДАТА**
   - label — название эпика (без пробелов, используй underscore)

   **Пример:** для 17 августа 2025 → `17_08_25_dtm_virtual_project_search.json`

2. **Структура файла**:

```
[
  {
    "section_id": 12345,
    "title": "User creation with valid data",
    "custom_preconds": "Пользователь авторизован в системе.\nОткрыта страница управления пользователями.",
    "type_id": 9,
    "priority_id": 2,
    "custom_automation": 1,
    "custom_automation_type": 0,
    "labels": [56, 87],
    "refs": "DS-8596",
    "custom_steps_separated": [
      {
        "content": "Заполнить все обязательные поля валидными данными и нажать кнопку \"Создать пользователя\"",
        "expected": "Пользователь создан, отображается сообщение об успехе"
      }
    ]
  }
]
```

3. **Самопроверка перед отправкой**:

- Все кейсы имеют четкие названия
- Шаги логически связаны
- Ожидаемые результаты конкретные
- Покрыты все сценарии из аналитики

4. **Отправь файл пользователю для ревью**
5. **СТОП**: дождись подтверждения или правок от пользователя. **НЕ ДОБАВЛЯЙ КЕЙСЫ В TESTRAIL** до получения команды!

---

### Шаг 7 — Добавление кейсов в TestRail (только по команде)

🔴 **ВАЖНО**: после создания MD файла работа **останавливается** и ожидается команда пользователя, например:

- “Выгрузи в TestRail” / “Добавь кейсы в TestRail” / “Залей в секцию `<SECTION_ID>`”

Что делать после команды:

1. Прочитай финальную версию `dd_mm_yy_label.json` из `results/02-test-cases-writting/`.
2. Проверь консистентность данных:
   - у каждого кейса есть обязательные поля (title, custom_preconds, steps/expected, refs, automation и т.д.)
   - корректно проставлены `type_id` и `priority_id`
   - шаги/expected парные (если используешь разделение шагов)
3. Добавь кейсы в TestRail через MCP:
   - по одному: `add_case(section_id=...)`
   - или пачкой: `add_cases(section_id=..., items=[...])`
4. В ответе пользователю дай:
   - сколько кейсов создано
   - список ID созданных кейсов

## Дополнительные рекомендации

### Частые ошибки и как их избежать

- **Слишком общие названия**: избегай названий типа "Проверка функции" — будь конкретен
- **Много действий в одном шаге**: один шаг = одно действие
- **Неконкретные ожидаемые результаты**: избегай "работает корректно" — опиши, что именно должно произойти
- **Отсутствие предусловий**: всегда указывай, что должно быть подготовлено перед тестом

### Примеры качественных формулировок

**❌ Плохо**: "Проверить авторизацию"
**✅ Хорошо**: "Verify user login with valid credentials displays dashboard"

**❌ Плохо**: "Нажать кнопку и проверить результат"  
**✅ Хорошо**: "Нажать кнопку 'Сохранить'" → "Отображается сообщение 'Данные успешно сохранены'"

### Контрольные вопросы для самопроверки

- Может ли другой тестировщик выполнить этот кейс без дополнительных вопросов?
- Покрывает ли набор кейсов все функциональности из аналитики?
- Есть ли проверки для основных негативных сценариев?
- Правильно ли выбраны приоритеты и типы кейсов?
