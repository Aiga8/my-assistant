## Задача (детали для LLM)

Ты — QA-аналитик.  
**Входные данные этапа (всегда от пользователя):**
- **Номер задачи** (например: QAA-6265).
- **Путь к спекам** — всегда внутри папки `tests from solo-rtm`. Пользователь может указать подпапку (например `trajectory`) или конкретные файлы; если указана только папка `tests from solo-rtm` — обработать все спеки в ней.

Сформировать **один** файл с описаниями автотестов в формате, пригодном для TestRail и для маппинга (Этап 3), с указанием названия каждого теста и пути к файлу в репозитории.

## Дополнительные требования

- **Один файл на выход:** если спеков несколько — все тесты из них объединить в один Markdown-файл со **сквозной нумерацией** (1, 2, 3, …).
- Добавить поля для дальнейшего заполнения TestRail ID и тегов.
- Сохранить результат в `results/01-autotest-coverage/`.
- Имя файла: `<номер_задачи>_<область>.md`, где `<область>` — по названию папки/файла (например: `QAA-6265_trajectory.md`).

## Правила

- Не упоминать моки, стабы, API, техническую реализацию, классы, селекторы и код.
- Не описывать внутреннюю реализацию.
- Не использовать формулировки про авторизацию, если она явно не проверяется.
- Описывать поведение приложения только с точки зрения пользователя.
- Формулировки должны быть нейтральными и обобщёнными.
- Описание должно строго соответствовать фактическим проверкам в тесте.
- Указывать только те проверки, которые действительно присутствуют в тесте (assert/expect).
- Если поведение не проверяется явно — не добавлять его в описание.
- Предусловия указывать только те, которые явно следуют из теста.
- Не объединять несколько логически разных сценариев в один кейс.
- Исходные spec-файлы не изменяются.

## Формат результата — Markdown (.md)

Структура для каждого теста:

---

### <номер>. Название кейса  
<краткое поведенческое название>

**TestRail:**
- ID: _(заполнить позже)_

**Предусловия:**
- <только явно необходимые условия>

**Шаги:**
1. <пользовательское действие>
2. <следующее действие>
3. <и т.д.>

**Ожидаемый результат:**
- <только реальные проверки из теста>

**Автотест:**
- Название теста: <имя test(...) из кода>
- Путь: <относительный путь к файлу в репозитории>

**Примечание (опционально):**
<если сценарий зависит от контекста>

---

После формирования Markdown-файла не добавлять пояснений от себя.
