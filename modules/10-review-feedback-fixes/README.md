# Инструкция для Cursor: Исправление замечаний по итогам ревью тестов в TestRail

## Быстрый старт (копипаст)

```
Открой и строго следуй инструкциям: modules/10-review-feedback-fixes/README.md

По секции <SECTION_ID> в TestRail покажи кейсы, вернувшиеся с ревью в статус Design и назначенные на меня. Выведи замечания ревьювера и помоги решить: исправлять или как поступить.
```

Пользователь всегда передаёт **ID главной секции** (SECTION_ID). Обход главной и всех подсекций, один итоговый отчёт. project_id и suite_id — см. `testrail-field-ids.md` (для DrillSpot обычно project_id = 5, suite_id = 5).

---

## 0. Цель модуля

После ревью тест-кейсов в TestRail часть кейсов ревьювер переводит в **Ready**, часть — в **Design** и назначает обратно на автора. В кейсах со статусом Design ревьювер оставляет **комментарии** (замечания). Задача модуля: собрать по указанной секции все такие кейсы, показать замечания и помочь автору принять решение — что исправить, что оспорить или оставить как есть.

---

## 1. Алгоритм работы

### Шаг 1 — Получение задачи

Пользователь указывает **SECTION_ID** главной секции. Обходим её и все подсекции, один итоговый отчёт. project_id, suite_id и ID автора («на меня») — в `testrail-field-ids.md`.

### Шаг 2 — Список секций для обхода

**Вариант A — через API (если главная секция в первых 250):**

1. Получить секции проекта: `get_sections(project_id, suite_id)` (suite_id из `testrail-field-ids.md` или уточнить).
2. Построить поддерево от главной секции: в множество section_id входят главная + все секции, у которых `parent_id` равен главной или уже входящей в множество (рекурсивно).

**Вариант B — через скрипт (рекомендуется при большом числе секций в проекте):**

API TestRail возвращает не более **250 секций** за один вызов. Если главная секция или её подсекции не попали в этот список, поддерево по `get_sections` построить нельзя.

**Скрипт для полного поддерева любой секции:**

```bash
node modules/01-autotest-coverage/testrail_export_section_subtree.mjs --root-section <SECTION_ID> --project <PROJECT_ID>
```

- **SECTION_ID** — ID главной секции из задачи пользователя.
- **PROJECT_ID** — ID проекта (для DrillSpot: 5, см. `testrail-field-ids.md`).

Скрипт с пагинацией получает все секции проекта, строит поддерево от `--root-section` и сохраняет:
- `results/01-autotest-coverage/testrail-section-<SECTION_ID>-subtree-cases.json` — в нём поле **subtreeSectionIds** (список всех section_id поддерева) и массив **cases** (все кейсы по поддереву).
- `results/01-autotest-coverage/testrail-section-<SECTION_ID>-subtree-cases.md` — человекочитаемый список секций и кейсов.

Дальше: либо использовать **subtreeSectionIds** из JSON и для каждого section_id вызывать `get_cases(project_id, section_id, ...)` (с пагинацией), либо отфильтровать кейсы из уже загруженного массива **cases** в JSON (по section_id, case_status_id, case_assignedto_id). Комментарии к кейсам в экспорте скрипта нет — для отображения замечаний ревьювера по каждому отфильтрованному кейсу вызывать `get_case(case_id)` (в ответе есть массив **comments**).

### Шаг 3 — Получение кейсов

1. Для каждой секции из списка: `get_cases(project_id, section_id, limit=250, offset=...)` с пагинацией. project_id — из задачи / `testrail-field-ids.md`.
2. Комментарии: в ответе `get_cases` массива comments может не быть. Для замечаний ревьювера по каждому отобранному кейсу вызывать **`get_case(case_id)`** — в ответе есть **`comments`**. В каждом комментарии: `user_id` (ревьювер), `comment` (HTML), `created_on`. **Запомнить `user_id` ревьювера** — после правок кейс вернуть ему.

### Шаг 4 — Фильтрация кейсов «вернувшихся с ревью»

Отобрать кейсы, у которых:
- **case_status_id = 2** (Design);
- **case_assignedto_id** = ID автора («на меня») из `testrail-field-ids.md`.

Это кейсы, которые ревьювер вернул на доработку и назначил на автора.

### Шаг 5 — Сводка по каждому кейсу

Для каждого отфильтрованного кейса вывести:
- **Case ID** и **Title**;
- **Замечания ревьювера:** текст из `comments` (при необходимости преобразовать HTML в читаемый текст);
- кратко: суть кейса (preconditions / steps / expected — по желанию, если нужно для контекста).

### Шаг 6 — Помощь в решении

По каждому кейсу с замечаниями предложить варианты:
- **Исправить** — что именно изменить в кейсе (шаги, ожидания, описание) с учётом замечания;
- **Не исправлять / оспорить** — если замечание некорректно или не согласен: сформулировать короткий ответ ревьюверу или решение «оставить как есть».

**Кейсы не удалять самому (через MCP/API).** Если ревьювер написал «удалить» — вывести это как рекомендацию; решение об удалении или переносе в другую секцию принимает пользователь сама.

Не вносить изменения в TestRail без явной команды пользователя. Сначала выдать сводку и рекомендации.

### Шаг 7 — Сохранение отчёта

Сохранить отчёт в каталог:
`results/10-review-feedback-fixes/`

Имя файла: `dd_mm_yy_section<SECTION_ID>_review_feedback.md` (SECTION_ID главной секции).

Пример: `19_02_26_section30048_review_feedback.md`

## 2. Формат вывода (пример)

```markdown
## Секция 30049 — кейсы, вернувшиеся с ревью (Design, назначены на меня)

**Всего в секции:** 9 кейсов  
**В статусе Ready после ревью:** 7  
**В статусе Design (на доработке):** 2  

---

### Кейс 353623 — "EDM Import unavailable with NO ACCESS in both GP and VP"

**Замечание ревьювера:**  
Не относится к функциональности импорта. Удалить

**Рекомендация:** Вывести замечание «удалить»; решение об удалении или переносе в другую секцию — за пользователем.

---

### Кейс 353622 — "Context menu UI matches design requirements"

**Замечание ревьювера:**  
"Активные пункты: синий цвет" — синего цвета нет.

**Рекомендация:** Исправить expected: убрать или заменить формулировку «синий цвет» на актуальное описание (например, цвет из дизайн-системы или «активный цвет по гайдам»). Либо уточнить у дизайна и обновить шаг.

---

## Итог

- Кейсы с замечаниями: 2
- Предлагаемые действия: [список]
```

## 3. Обновление кейсов в TestRail

После того как пользователь решит, что править:

- Вносить правки через MCP TestRail: `update_case` / `update_case_fields` (в т.ч. `custom_steps_separated` для шагов и ожиданий).
- **Вернуть кейс ревьюверу** после исправлений: того же человека, кто оставил замечание (ревьювер не всегда один и тот же). При начале работы по кейсу запомнить **`user_id` из комментария** — это ревьювер. После правок выставить:
  - **case_status_id: 3** (Review);
  - **case_assignedto_id** = `user_id` ревьювера (из комментария к кейсу).
  Имена/ID пользователей при необходимости — в `testrail-field-ids.md`.
- Если кейс остаётся у автора в статусе Ready (без отправки на ревью): **case_status_id: 1** (Ready). При любом обновлении кейса нужно явно передавать нужный статус, иначе TestRail может сбросить его в Design (см. `operations.md`).
- Не удалять кейсы через MCP. Если ревьювер просит удалить — вывести как рекомендацию; удаление выполняет пользователь сама при необходимости.
