## 0. Цель модуля

По запросу пользователя создать в Jira задачу: вариант A — на написание автотестов и линковку с TestRail; вариант B — на фиксы автотестов (без линковки); вариант C — на создание/актуализацию тест-кейсов в TestRail, при необходимости с линковкой на автотесты. Формат задачи описан ниже.

---

## 1. Входные данные

Пользователь указывает:

- **Область/фича** — короткое название (например: Trace list, Format ribbons, Left Panel).
- **Секция TestRail** — `section_id` (для вариантов A и C; project_id для DrillSpot/RTM: **5**).
- **Спек** — путь к файлу спеки (для варианта B обязателен; для варианта C — опционально, если нужна линковка с автотестами).
- **Тип задачи** (вариант A — линковка и написание; вариант B — фиксы; вариант C — создание/актуализация кейсов в TestRail, при необходимости линковка).

---

## 2. Шаблон задачи в Jira

### Summary (тема)

- **Вариант A (линковка и написание):** `[UI DS] <Область/фича>: Add Playwright E2E tests and align mapping with TestRail`
- **Вариант B (фиксы):** `[UI DS] <Область/фича>: fix tests`
- **Вариант C (TestRail):** `[UI DS] <Область/фича>: Create/update TestRail cases (align mapping if needed)`

Примеры:  
`[UI DS] Trace list: align mapping with TestRail`  
`[UI DS] Format ribbons: align mapping with TestRail`  
`[UI DS] Trace list: Create/update TestRail cases (align mapping if needed)`

### Description (описание)

**Вариант A — линковка и написание:**

```
Нужно покрыть автотестами функциональность секции <Область/фича> (id <section_id>), чтобы стабильно проверять ключевые пользовательские сценарии и предотвратить регрессии.

*Acceptance Criteria*

* Добавлены E2E тесты Playwright для основных сценариев <Область/фича> (спек: <путь к спеки>)
* Сценарии сопоставлены с кейсами в TestRail секции "<Область/фича>"
* Для каждого кейса добавлен референс на автотест (spec path + test name). Подход — QAA-6117
```

**Вариант B — фиксы:**

```
По Playwright-тестам, относящимся к <Область/фича>, выполнить фиксы падающих/неактуальных тестов.

*Критерии приёмки*
* Исправить падающие/нестабильные тесты по <Область/фича> (спек: <путь к спеки>)
* Привести шаги и ожидания в соответствие с текущим UI/логикой
* Падающие тесты исправлены и проходят стабильно
```

**Вариант C — создание/актуализация кейсов в TestRail (при необходимости линковка):**

```
Создать или актуализировать тест-кейсы в TestRail по области <Область/фича> (секция id <section_id>). При необходимости проставить связи между кейсами и автотестами (линковка).

*Критерии приёмки*
* Тест-кейсы в секции TestRail "<Область/фича>" созданы или обновлены (актуальны по аналитике/фиче)
* При необходимости: сценарии сопоставлены с автотестами (спек: <путь к спеки>), для кейсов добавлен референс на автотест (spec path + test name). Подход — QAA-6117
```

В описании подставь фактические: `<Область/фича>`, для варианта A — `<section_id>` и путь к спеки, для варианта B — путь к спеки, для варианта C — `<section_id>` и при необходимости путь к спеки.

---

## 3. Создание задачи через MCP

Создание задачи выполняй через **MCP Atlassian (Jira)** — инструмент `jira_create_issue`.

Перед вызовом проверь схему: `mcps/user-mcp-atlassian/tools/jira_create_issue.json`. Обязательные параметры: `project_key`, `summary`, `issue_type`.

Рекомендуемые значения:

- **project_key** — **QAA** (проект задач на автотесты).
- **issue_type** — **Task**.
- **summary** — из шаблона выше.
- **description** — из шаблона с подставленными данными.
- **assignee** — **на меня**: Aigerim Suleimenova (или `a.suleimenova@rogii.com`).
- **additional_fields** — обязательно:
  - **Labels:** `"labels": ["QAA"]`
  - **Dev Team:** `"customfield_10739": ["DrillSpot"]`

После создания верни пользователю ключ созданной задачи и ссылку (например `https://starsteer.atlassian.net/browse/QAA-XXXX`).

---

## 4. Алгоритм работы

1. **Получить входные данные** от пользователя: область/фича, тип задачи (вариант A / B / C); для A — section_id и путь к спеке, для B — путь к спеке, для C — section_id и опционально путь к спеке (если линковка). При отсутствии project_key использовать QAA.
2. **Сформировать Summary и Description** по шаблонам из раздела 2, подставив указанные область, section_id и путь к спеки.
3. **Вызвать** `jira_create_issue` (MCP user-mcp-atlassian) с полученными полями.
4. **Вернуть** пользователю ключ задачи и ссылку на неё в Jira.

Если пользователь просит только **текст задачи** (без создания в Jira) — сформируй и выведи только Summary + Description по шаблонам.
